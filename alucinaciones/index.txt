<p>Las alucinaciones no son todas iguales. Un modelo que inventa una función de una librería es un problema diferente a uno que razona incorrectamente sobre lógica de negocio. La detección y el manejo cambian según el tipo.</p>

<h2>Tipo 1: Alucinación de API / librería</h2>
<p>El modelo inventa funciones, métodos o parámetros que no existen. Es el tipo más fácil de detectar: el código no compila o arroja un error inmediato en runtime.</p>
<p><strong>Causas:</strong> APIs cambiaron después del cutoff del modelo, librerías nicho con poca documentación pública, confusión entre librerías similares.</p>
<p><strong>Detección:</strong> compilación + linter. Si usas TypeScript, la mayoría aparece como error de tipos.</p>
<p><strong>Prevención:</strong> incluir la versión exacta de la librería en el prompt, pegar la firma real de la función cuando la conoces.</p>

<h2>Tipo 2: Alucinación factual</h2>
<p>El modelo afirma hechos incorrectos con confianza: "PostgreSQL no soporta X", "el estándar dice Y", "esta vulnerabilidad fue parchada en la versión Z". Son peligrosas porque son plausibles.</p>
<p><strong>Causas:</strong> información desactualizada respecto al cutoff, mezcla de fuentes contradictorias, generalización incorrecta de casos similares.</p>
<p><strong>Detección:</strong> requiere verificación externa. El modelo no va a decirte cuándo está inventando un hecho.</p>
<p><strong>Cuándo verificar siempre:</strong> afirmaciones sobre seguridad, compliance, comportamiento de protocolos de red, garantías de bases de datos.</p>

<h2>Tipo 3: Razonamiento incorrecto pero plausible</h2>
<p>El más peligroso. El código es sintácticamente correcto, usa APIs que existen, y aun así hace algo incorrecto. La lógica tiene un error sutil que solo aparece en casos edge o bajo ciertas condiciones.</p>
<p><strong>Ejemplos:</strong> off-by-one en loops, comparaciones incorrectas en condiciones de borde, asunciones sobre el orden de operaciones async.</p>
<p><strong>Detección:</strong> tests, revisión manual, mutation testing.</p>

<h2>Construir hábitos de verificación en el equipo</h2>
<p>El problema con las alucinaciones es que no hay un síntoma visible hasta que el código falla. La prevención tiene que ser proactiva:</p>
<ul>
  <li><strong>Define zonas de verificación obligatoria</strong> en tu CLAUDE.md o en las normas del equipo. Áreas donde el código generado siempre pasa por un segundo par de ojos humanos o por tests específicos.</li>
  <li><strong>Pide referencias.</strong> "¿En qué documentación oficial basas esta afirmación?" obliga al modelo a ser más cuidadoso o a admitir incertidumbre.</li>
  <li><strong>Usá el modelo para verificarse a sí mismo.</strong> "Revisá este código que generaste anteriormente buscando errores lógicos y asunciones incorrectas." No es perfecto, pero detecta algunos errores.</li>
  <li><strong>Agregá contexto de versión explícito.</strong> "Estoy usando PostgreSQL 16, Node 22, TypeScript 5.5." Reduce alucinaciones de compatibilidad.</li>
</ul>

<h2>La confianza calibrada</h2>
<p>El objetivo no es desconfiar de todo lo que genera la IA —eso elimina el beneficio. El objetivo es tener confianza calibrada: alta en tareas mecánicas bien definidas, baja en afirmaciones sobre el mundo real o lógica de dominio compleja.</p>
