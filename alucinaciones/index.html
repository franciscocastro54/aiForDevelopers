Revisión y calidad

Alucinaciones y verificación

Tipos de alucinaciones, cómo detectar cada una y qué verificar siempre manualmente.

Las alucinaciones no son todas iguales. Un modelo que inventa una función de una librería es un problema diferente a uno que razona incorrectamente sobre lógica de negocio. La detección y el manejo cambian según el tipo.

Tipo 1: Alucinación de API / librería

El modelo inventa funciones, métodos o parámetros que no existen. Es el tipo más fácil de detectar: el código no compila o arroja un error inmediato en runtime.

Causas: APIs cambiaron después del cutoff del modelo, librerías nicho con poca documentación pública, confusión entre librerías similares.

Detección: compilación + linter. Si usás TypeScript, la mayoría aparece como error de tipos.

Prevención: incluir la versión exacta de la librería en el prompt, pegar la firma real de la función cuando la conocés.

Tipo 2: Alucinación factual

El modelo afirma hechos incorrectos con confianza: "PostgreSQL no soporta X", "el estándar dice Y", "esta vulnerabilidad fue parchada en la versión Z". Son peligrosas porque son plausibles.

Causas: información desactualizada respecto al cutoff, mezcla de fuentes contradictorias, generalización incorrecta de casos similares.

Detección: requiere verificación externa. El modelo no va a decirte cuándo está inventando un hecho.

Cuándo verificar siempre: afirmaciones sobre seguridad, compliance, comportamiento de protocolos de red, garantías de bases de datos.

Tipo 3: Razonamiento incorrecto pero plausible

El más peligroso. El código es sintácticamente correcto, usa APIs que existen, y aun así hace algo incorrecto. La lógica tiene un error sutil que solo aparece en casos edge o bajo ciertas condiciones.

Ejemplos: off-by-one en loops, comparaciones incorrectas en condiciones de borde, asunciones sobre el orden de operaciones async.

Detección: tests, revisión manual, mutation testing.

🚨 Áreas de verificación obligatoria

Criptografía, autenticación, autorización, validación de pagos, operaciones destructivas irreversibles. En estas áreas, siempre verificá contra la documentación oficial o un experto, sin importar qué tan convincente suene la IA.

Construir hábitos de verificación en el equipo

El problema con las alucinaciones es que no hay un síntoma visible hasta que el código falla. La prevención tiene que ser proactiva:

  • Define zonas de verificación obligatoria en tu CLAUDE.md o en las normas del equipo. Áreas donde el código generado siempre pasa por un segundo par de ojos humanos o por tests específicos.
  • Pedí referencias. "¿En qué documentación oficial basás esta afirmación?" obliga al modelo a ser más cuidadoso o a admitir incertidumbre.
  • Usá el modelo para verificarse a sí mismo. "Revisá este código que generaste anteriormente buscando errores lógicos y asunciones incorrectas." No es perfecto, pero detecta algunos errores.
  • Agregá contexto de versión explícito. "Estoy usando PostgreSQL 16, Node 22, TypeScript 5.5." Reduce alucinaciones de compatibilidad.

La confianza calibrada

El objetivo no es desconfiar de todo lo que genera la IA —eso elimina el beneficio. El objetivo es tener confianza calibrada: alta en tareas mecánicas bien definidas, baja en afirmaciones sobre el mundo real o lógica de dominio compleja.

💡 Señal de incertidumbre

Los buenos modelos expresan incertidumbre cuando no saben. Frases como "no estoy seguro de esto, verificá la documentación" son buenas señales. Si el modelo siempre suena igual de confiado, aumentá el escepticismo.
