Agentes y autonomía

La diferencia entre copiloto y agente, cuándo dar autonomía y qué barreras poner.

La distinción entre copiloto y agente no es de capacidades, es de control. Un copiloto sugiere y el desarrollador aprueba cada acción. Un agente ejecuta una secuencia de acciones —editar archivos, correr comandos, abrir PRs— con supervisión humana mínima o nula. La diferencia en implicancias de riesgo es enorme.

El espectro de autonomía

  1. Autocompletado: sugerencia de la siguiente línea. El humano activa con Tab.
  2. Generación de bloque: implementa una función completa. El humano revisa y aplica.
  3. Edición de archivo: modifica un archivo entero con diff. El humano aprueba el diff.
  4. Multi-archivo: edita varios archivos, corre tests, muestra resultado. El humano aprueba antes de commitear.
  5. Agente con loop: itera hasta que los tests pasan, commitea, abre PR. Aprobación humana solo en puntos de control definidos.
  6. Agente autónomo: ejecuta tareas de principio a fin sin intervención. Solo se activa post-hoc.

Herramientas actuales por posición en el espectro

  • GitHub Copilot: nivel 1-2
  • Cursor / Windsurf (modo normal): nivel 2-3
  • Claude Code / Cursor (modo agente): nivel 3-5
  • Devin, SWE-agent: nivel 5-6

⚠️ El nivel de autonomía correcto depende de la reversibilidad

Más autonomía es aceptable cuando las acciones son reversibles (editar archivos en una feature branch) y riesgoso cuando no lo son (pushear a main, ejecutar migraciones de DB, enviar notificaciones a usuarios).

Barreras para agentes con autonomía alta

A medida que le das más autonomía a un agente, las barreras tienen que ser proporcionales:

  • Branch aislada: el agente trabaja en su propia branch, nunca en main
  • Sin push automático: el agente puede commitear localmente, pero el push lo hace un humano
  • Tests como gate: el agente no puede declarar éxito si los tests fallan
  • Sin acceso a credenciales de producción
  • Logs de todo lo que ejecutó: poder auditar qué comandos corrió
  • Límite de tiempo: si el agente lleva más de X minutos, se pausa y reporta estado

Cuándo es buena idea usar un agente autónomo

  • Tareas mecánicas bien definidas con criterio de éxito claro (migrar una API a nueva versión, renombrar un patrón en todo el codebase)
  • Generación de tests para código existente donde hay especificación clara
  • Actualizaciones de dependencias con suite de tests confiable
  • Generación de boilerplate repetitivo siguiendo un patrón probado

Cuándo no es buena idea

  • Lógica de negocio con reglas implícitas que solo el equipo conoce
  • Cambios en sistemas críticos sin cobertura de tests alta
  • Cualquier acción con efectos fuera del repositorio (emails, APIs externas, bases de datos de producción)
  • Tareas donde "funciona" no es el criterio correcto (UX, accesibilidad, performance)

El futuro: agentic engineering

La dirección de la industria es clara: los agentes van a ejecutar más tareas completas de forma autónoma. El rol del desarrollador va a evolucionar hacia definir especificaciones precisas, diseñar las barreras correctas, y auditar el trabajo del agente. Las habilidades de ingeniería de software clásicas —arquitectura, testing, razonamiento sobre sistemas— se vuelven más valiosas, no menos, porque son las que el agente no puede suplir.
