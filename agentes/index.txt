<p>La distinción entre copiloto y agente no es de capacidades, es de control. Un copiloto sugiere y el desarrollador aprueba cada acción. Un agente ejecuta una secuencia de acciones —editar archivos, correr comandos, abrir PRs— con supervisión humana mínima o nula. La diferencia en implicancias de riesgo es enorme.</p>

<h2>El espectro de autonomía</h2>
<ol>
  <li><strong>Autocompletado:</strong> sugerencia de la siguiente línea. El humano activa con Tab.</li>
  <li><strong>Generación de bloque:</strong> implementa una función completa. El humano revisa y aplica.</li>
  <li><strong>Edición de archivo:</strong> modifica un archivo entero con diff. El humano aprueba el diff.</li>
  <li><strong>Multi-archivo:</strong> edita varios archivos, corre tests, muestra resultado. El humano aprueba antes de commitear.</li>
  <li><strong>Agente con loop:</strong> itera hasta que los tests pasan, commitea, abre PR. Aprobación humana solo en puntos de control definidos.</li>
  <li><strong>Agente autónomo:</strong> ejecuta tareas de principio a fin sin intervención. Solo se activa post-hoc.</li>
</ol>

<h2>Herramientas actuales por posición en el espectro</h2>
<ul>
  <li><strong>GitHub Copilot:</strong> nivel 1-2</li>
  <li><strong>Cursor / Windsurf (modo normal):</strong> nivel 2-3</li>
  <li><strong>Claude Code / Cursor (modo agente):</strong> nivel 3-5</li>
  <li><strong>Devin, SWE-agent:</strong> nivel 5-6</li>
</ul>

<h2>Barreras para agentes con autonomía alta</h2>
<p>A medida que le das más autonomía a un agente, las barreras tienen que ser proporcionales:</p>
<ul>
  <li><strong>Branch aislada:</strong> el agente trabaja en su propia branch, nunca en main</li>
  <li><strong>Sin push automático:</strong> el agente puede commitear localmente, pero el push lo hace un humano</li>
  <li><strong>Tests como gate:</strong> el agente no puede declarar éxito si los tests fallan</li>
  <li><strong>Sin acceso a credenciales de producción</strong></li>
  <li><strong>Logs de todo lo que ejecutó:</strong> poder auditar qué comandos corrió</li>
  <li><strong>Límite de tiempo:</strong> si el agente lleva más de X minutos, se pausa y reporta estado</li>
</ul>

<h2>Cuándo es buena idea usar un agente autónomo</h2>
<ul>
  <li>Tareas mecánicas bien definidas con criterio de éxito claro (migrar una API a nueva versión, renombrar un patrón en todo el codebase)</li>
  <li>Generación de tests para código existente donde hay especificación clara</li>
  <li>Actualizaciones de dependencias con suite de tests confiable</li>
  <li>Generación de boilerplate repetitivo siguiendo un patrón probado</li>
</ul>

<h2>Cuándo no es buena idea</h2>
<ul>
  <li>Lógica de negocio con reglas implícitas que solo el equipo conoce</li>
  <li>Cambios en sistemas críticos sin cobertura de tests alta</li>
  <li>Cualquier acción con efectos fuera del repositorio (emails, APIs externas, bases de datos de producción)</li>
  <li>Tareas donde "funciona" no es el criterio correcto (UX, accesibilidad, performance)</li>
</ul>

<h2>El futuro: agentic engineering</h2>
<p>La dirección de la industria es clara: los agentes van a ejecutar más tareas completas de forma autónoma. El rol del desarrollador va a evolucionar hacia definir especificaciones precisas, diseñar las barreras correctas, y auditar el trabajo del agente. Las habilidades de ingeniería de software clásicas —arquitectura, testing, razonamiento sobre sistemas— se vuelven más valiosas, no menos, porque son las que el agente no puede suplir.</p>
