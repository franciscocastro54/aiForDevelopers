<h2>Las tres palancas de reducción de costo</h2>

<h3>1. Modelo correcto para cada tarea</h3>
<p>El error más caro: usar Opus para todo. Haiku cuesta 20x menos que Opus y es perfectamente capaz para boilerplate, renombrado, generación de tests simples, resúmenes. Mapear tareas → modelo óptimo puede reducir el gasto a la mitad sin perder calidad.</p>

<h3>2. Caché de prompts</h3>
<p>Si el inicio de tu contexto es estable —system prompt largo, documentación de referencia, CLAUDE.md— los proveedores cobran los tokens cacheados a ~10% del precio base. Para sesiones largas con contexto fijo, el ahorro puede ser del 40-60% del costo total.</p>

<h3>3. Contexto mínimo suficiente</h3>
<p>Cada archivo que agregás al contexto se cobra. Enviar el repositorio completo cuando necesitás una función es como pagar un vuelo a Europa para ir al supermercado. El contexto tiene que ser proporcional a la tarea.</p>

<h2>Medir ROI real</h2>
<p>El costo de la IA es visible (factura del proveedor). El beneficio es menos obvio. Métricas útiles:</p>
<ul>
  <li><strong>Tiempo en tareas mecánicas:</strong> ¿cuánto tiempo del sprint va en boilerplate, tests básicos, documentación? Si ese tiempo baja, hay ROI.</li>
  <li><strong>Velocidad de onboarding:</strong> ¿los developers nuevos necesitan menos tiempo para ser productivos?</li>
  <li><strong>Bugs en producción:</strong> si la IA ayuda a generar más tests, ¿bajó la tasa de bugs?</li>
  <li><strong>Costo por feature:</strong> comparar la inversión de tiempo en una feature antes y después de adoptar IA.</li>
</ul>

<h2>Presupuesto por equipo</h2>
<p>Una heurística práctica para equipos sin uso de APIs internas: los planes de suscripción (Claude Pro, GitHub Copilot Individual) son suficientes para la mayoría de los casos de uso interactivos. Las APIs directas tienen sentido cuando se automatizan pipelines o se construyen herramientas internas. No pagues por APIs si usás herramientas con suscripción.</p>
