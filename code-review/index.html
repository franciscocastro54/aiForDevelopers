Revisión y calidad

Code review con IA

Cómo cambia el rol del reviewer cuando el autor usó IA, y cómo evitar que el review también se tercerice.

Code review cuando el autor usó IA es un problema distinto a "revisar código generado por IA tú mismo". En el primer caso, hay un ser humano que debería entender el código que está subiendo. En el segundo, la responsabilidad de comprensión está distribuida de una forma que puede crear puntos ciegos.

¿Cambia el rol del reviewer?

Sí, en un aspecto clave: cuando el autor usó IA sin revisión crítica profunda, la probabilidad de código "plausible pero incorrecto" es más alta que con código escrito a mano. La IA produce código que parece correcto —buena sintaxis, nombres razonables, patrones reconocibles— pero puede tener suposiciones incorrectas sobre el dominio.

El reviewer tiene que subir el nivel de escepticismo ante código que parece "demasiado limpio" o "demasiado genérico" para el problema específico.

Señales de código generado sin revisión suficiente

  • Manejo de errores genérico que no refleja los errores reales del sistema
  • Comentarios que describen el código en vez de explicar el porqué
  • Imports de librerías no usadas o de APIs en versiones incorrectas
  • Tipos any o casts que indican que la IA no resolvió el problema de tipos
  • Lógica de negocio que es técnicamente correcta pero ignora reglas del dominio
  • Tests que espejean la implementación en lugar de verificar comportamiento

ℹ️ ¿Pedir disclosure?

Algunos equipos requieren que el autor indique en el PR si usó IA y qué revisó manualmente. Esto no es para juzgar, sino para calibrar el nivel de revisión necesario. Un PR donde el autor dice "generé con IA y revisé línea por línea" es diferente a "generé y compiló".

Evitar que el reviewer tercerice a IA

El anti-patrón emerge rápido: autor usa IA para escribir el código, reviewer usa IA para revisarlo, nadie realmente leyó el código. El resultado es un PR aprobado por dos modelos de lenguaje y ningún humano.

La IA es una herramienta válida durante el review —para buscar patrones, verificar lógica compleja, generar casos de test adicionales— pero el reviewer tiene que tener su propio modelo mental del cambio antes de usarla como auxiliar.

Cómo usar IA en el review de forma legítima

  • Después de leer el código tú: "¿Ves algún caso edge que yo podría haber pasado por alto?" (con el código pegado)
  • Para verificar lógica compleja: "Este algoritmo debería producir estos outputs para estos inputs. ¿Estás de acuerdo?"
  • Para sugerir tests adicionales: "¿Qué casos no están cubiertos por estos tests?"
  • Para entender código que no conoces: "Explícame qué hace esta sección para que pueda revisarla con criterio"

Responsabilidad y ownership

Un principio útil para el equipo: quien aprueba un PR es responsable del código, independientemente de si fue escrito por un humano o una IA. Esto pone el incentivo correcto: revisar bien.

Sugerencias para normas de equipo

  • El autor es responsable de entender cada línea que sube, sin importar cómo la generó
  • El reviewer puede pedir que el autor explique cualquier sección durante la review
  • Los tests son obligatorios independientemente del origen del código
  • Si el autor no puede explicar una sección, debe reescribirla hasta entenderla
