<p>Code review cuando el autor usó IA es un problema distinto a "revisar código generado por IA tú mismo". En el primer caso, hay un ser humano que debería entender el código que está subiendo. En el segundo, la responsabilidad de comprensión está distribuida de una forma que puede crear puntos ciegos.</p>

<h2>¿Cambia el rol del reviewer?</h2>
<p>Sí, en un aspecto clave: cuando el autor usó IA sin revisión crítica profunda, la probabilidad de código "plausible pero incorrecto" es más alta que con código escrito a mano. La IA produce código que <em>parece</em> correcto —buena sintaxis, nombres razonables, patrones reconocibles— pero puede tener suposiciones incorrectas sobre el dominio.</p>

<p>El reviewer tiene que subir el nivel de escepticismo ante código que parece "demasiado limpio" o "demasiado genérico" para el problema específico.</p>

<h2>Señales de código generado sin revisión suficiente</h2>
<ul>
  <li>Manejo de errores genérico que no refleja los errores reales del sistema</li>
  <li>Comentarios que describen el código en vez de explicar el porqué</li>
  <li>Imports de librerías no usadas o de APIs en versiones incorrectas</li>
  <li>Tipos <code>any</code> o casts que indican que la IA no resolvió el problema de tipos</li>
  <li>Lógica de negocio que es técnicamente correcta pero ignora reglas del dominio</li>
  <li>Tests que espejean la implementación en lugar de verificar comportamiento</li>
</ul>

<h2>Evitar que el reviewer tercerice a IA</h2>
<p>El anti-patrón emerge rápido: autor usa IA para escribir el código, reviewer usa IA para revisarlo, nadie realmente leyó el código. El resultado es un PR aprobado por dos modelos de lenguaje y ningún humano.</p>

<p>La IA es una herramienta válida durante el review —para buscar patrones, verificar lógica compleja, generar casos de test adicionales— pero el reviewer tiene que tener su propio modelo mental del cambio antes de usarla como auxiliar.</p>

<h2>Cómo usar IA en el review de forma legítima</h2>
<ul>
  <li><strong>Después de leer el código tú:</strong> "¿Ves algún caso edge que yo podría haber pasado por alto?" (con el código pegado)</li>
  <li><strong>Para verificar lógica compleja:</strong> "Este algoritmo debería producir estos outputs para estos inputs. ¿Estás de acuerdo?" </li>
  <li><strong>Para sugerir tests adicionales:</strong> "¿Qué casos no están cubiertos por estos tests?"</li>
  <li><strong>Para entender código que no conoces:</strong> "Explícame qué hace esta sección para que pueda revisarla con criterio"</li>
</ul>

<h2>Responsabilidad y ownership</h2>
<p>Un principio útil para el equipo: quien aprueba un PR es responsable del código, independientemente de si fue escrito por un humano o una IA. Esto pone el incentivo correcto: revisar bien.</p>

<h2>Sugerencias para normas de equipo</h2>
<ul>
  <li>El autor es responsable de entender cada línea que sube, sin importar cómo la generó</li>
  <li>El reviewer puede pedir que el autor explique cualquier sección durante la review</li>
  <li>Los tests son obligatorios independientemente del origen del código</li>
  <li>Si el autor no puede explicar una sección, debe reescribirla hasta entenderla</li>
</ul>
