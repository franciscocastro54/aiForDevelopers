Código con IA

Debugging

Cómo describir bugs, qué contexto incluir y cómo iterar cuando la primera solución no funciona.

Debuggear con IA es, en esencia, un problema de comunicación: cuánto contexto podés transferirle al modelo antes de que sugiera algo. El desarrollador que aprende a describir bugs de forma precisa obtiene diagnósticos precisos. El que pega el stacktrace sin contexto recibe soluciones genéricas.

Los cinco elementos de un bug report efectivo para IA

1. Comportamiento esperado vs observado

No "tengo un error", sino "esperaba que la función devolviera X, devolvió Y". Ser explícito sobre la diferencia entre lo que el sistema debería hacer y lo que hace.

2. Condiciones de reproducción

¿Ocurre siempre? ¿Solo con ciertos inputs? ¿Solo en producción? ¿Solo con sesión iniciada? Las condiciones de reproducción son la mitad del diagnóstico.

3. Stacktrace completo

No el primer frame. No un screenshot. El stacktrace completo como texto, con los frames de tu código y los de las dependencias relevantes.

4. El código relevante

La función donde ocurre, sus dependencias directas, los tipos involucrados. No todo el repositorio, pero sí el contexto suficiente para que la IA entienda qué interactúa con qué.

5. Qué ya probaste

Si ya intentaste algo y no funcionó, decirlo evita que la IA sugiera lo mismo.

Plantilla de bug report para IA

Cuando la primera solución no funciona

El error más común: aplicar la sugerencia sin entenderla, ver que no funciona, y pedir "otra solución" sin más contexto. Esto genera un loop de soluciones aleatorias.

El flujo correcto cuando falla la primera sugerencia:

  1. Describí exactamente qué cambió al aplicar la solución
  2. Si el error cambió, pegá el nuevo error
  3. Si el error es el mismo, confirmalo explícitamente
  4. Agregá cualquier nueva información que descubriste al intentar la solución

Debugging de problemas de concurrencia

Los bugs de concurrencia son el caso donde la IA es menos confiable: no puede "ver" el timing, y sus sugerencias tienden a ser soluciones estándar (mutexes, locks) sin entender si aplican a tu caso. Para estos casos:

  • Describí el modelo de concurrencia de tu sistema (threads, event loop, async/await)
  • Explicá qué recursos compartidos están involucrados
  • Describí el orden de operaciones en el que se reproduce
  • Verificá cualquier solución propuesta contra tu modelo mental antes de aplicarla

💡 Debugging inverso

Si no encontrás el bug, pedile a la IA que genere una lista de todas las formas en que esa función podría fallar dado el contexto que le diste. A veces identificar el espacio de posibilidades es más útil que buscar la causa específica.
