Código con IA

Testing

Generar, revisar y mantener tests con IA sin sacrificar su valor real.

La IA genera tests rápido. El problema es que genera exactamente lo que le pedís: si le pedís tests para una función, va a escribir tests que prueban esa función —no tests que prueban que el sistema funciona. La distinción importa.

TDD con IA: el orden correcto

El mayor beneficio de usar IA con TDD no es que genere los tests —es que los tests actúan como especificación, y la IA puede implementar contra esa especificación.

  1. Escribes (tú o con ayuda de la IA) los tests que describen el comportamiento esperado
  2. Los tests fallan (porque no hay implementación)
  3. Le pedís a la IA que implemente el código para hacer pasar esos tests
  4. La IA no puede "hacer trampa" cambiando los tests porque tú los controlas

Este flujo invierte el control: la IA implementa contra tu especificación, no al revés.

Qué tests genera bien la IA

  • Happy path para funciones con comportamiento claro
  • Tests de validación de input (null, undefined, tipos incorrectos)
  • Tests de serialización/deserialización
  • Tests de integración cuando le das el contrato de la API

Qué tests genera mal

  • Tests que verifican comportamiento de negocio implícito que no está en el código
  • Tests de concurrencia y condiciones de carrera
  • Tests de performance con umbrales reales
  • Tests de seguridad (tiende a testear el happy path de autenticación, no los bypasses)
  • Tests que van a fallar en el futuro cuando cambien dependencias

🚨 Anti-patrón crítico

La IA tiende a generar tests que espejean la implementación en vez de testear el comportamiento. Si refactorizas la implementación y los tests siguen pasando sin cambios, bien. Si necesitas cambiar los tests cada vez que cambias la implementación, los tests están testeando el cómo, no el qué.

Prompt para generar tests de calidad

Cobertura vs calidad

La IA puede llevarte a 90% de cobertura en horas. El problema: cobertura alta no garantiza tests de calidad. Un test que ejecuta una línea sin verificar su resultado cuenta como cobertura.

Métrica más útil que cobertura: mutation testing. Si mutás la implementación (cambiás un > por >=, invertís una condición), los tests deben fallar. La IA puede ayudarte a identificar qué mutaciones sobrevivirían a tu suite actual.

Mantenibilidad de tests generados

Los tests generados por IA a veces usan patrones verbosos o repiten setup que podría estar en fixtures. Revisalos con el mismo criterio que cualquier código: claridad, DRY razonable, nombres que explican la intención.
