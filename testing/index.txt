<p>La IA genera tests rápido. El problema es que genera exactamente lo que le pedís: si le pedís tests para una función, va a escribir tests que prueban esa función —no tests que prueban que el sistema funciona. La distinción importa.</p>

<h2>TDD con IA: el orden correcto</h2>
<p>El mayor beneficio de usar IA con TDD no es que genere los tests —es que los tests actúan como especificación, y la IA puede implementar contra esa especificación.</p>
<ol>
  <li>Escribes (tú o con ayuda de la IA) los tests que describen el comportamiento esperado</li>
  <li>Los tests fallan (porque no hay implementación)</li>
  <li>Le pedís a la IA que implemente el código para hacer pasar esos tests</li>
  <li>La IA no puede "hacer trampa" cambiando los tests porque tú los controlas</li>
</ol>
<p>Este flujo invierte el control: la IA implementa contra tu especificación, no al revés.</p>

<h2>Qué tests genera bien la IA</h2>
<ul>
  <li>Happy path para funciones con comportamiento claro</li>
  <li>Tests de validación de input (null, undefined, tipos incorrectos)</li>
  <li>Tests de serialización/deserialización</li>
  <li>Tests de integración cuando le das el contrato de la API</li>
</ul>

<h2>Qué tests genera mal</h2>
<ul>
  <li>Tests que verifican comportamiento de negocio implícito que no está en el código</li>
  <li>Tests de concurrencia y condiciones de carrera</li>
  <li>Tests de performance con umbrales reales</li>
  <li>Tests de seguridad (tiende a testear el happy path de autenticación, no los bypasses)</li>
  <li>Tests que van a fallar en el futuro cuando cambien dependencias</li>
</ul>
