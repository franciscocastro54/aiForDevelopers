Escala y legacy

Codebases grandes y legacy

Estrategias para dar contexto suficiente cuando el codebase supera la ventana de contexto del modelo.

La mayoría de los tutoriales de IA asumen proyectos chicos. El problema real empieza cuando el codebase tiene 500k líneas, convenciones internas no documentadas, y código legacy que nadie quiere tocar. Ahí es donde las técnicas estándar fallan y necesitas estrategias específicas.

El problema de escala

Un context window de 200k tokens suena amplio hasta que lo pones en perspectiva: es aproximadamente 150,000 líneas de código, o un proyecto mediano. Un monorepo grande puede tener 10x eso. La IA no puede "ver" todo el codebase a la vez —necesita que tú elijas qué ver.

Herramientas con indexación de codebase

Estas herramientas generan embeddings del codebase y recuperan fragmentos relevantes automáticamente cuando haces una pregunta, en lugar de requerir que tú elijas manualmente qué incluir:

  • Cursor: indexación automática, búsqueda semántica del codebase, integración con codebase privado
  • Claude Code: puede indexar y buscar en el directorio de trabajo
  • GitHub Copilot (Enterprise): acceso a todo el repositorio privado
  • Codeium: alternativa open-source-friendly

Estrategias para código legacy

Sin tests: empezá por caracterization tests

Antes de refactorizar código legacy sin tests, generá tests que capturan el comportamiento actual —aunque ese comportamiento sea incorrecto. Son tests de "caracterización": documentan qué hace el sistema, no qué debería hacer. La IA es excelente para esto dado el código.

Convenciones internas: documentalas explícitamente

Si tu codebase tiene convenciones no escritas —"siempre usamos el patrón X para Y", "nunca hagas Z en este módulo"— documentalas en CLAUDE.md o en comentarios de módulo. La IA no puede inferir convenciones implícitas que no están en el código visible.

Ejercicio útil: pedile a la IA que analice 5-10 archivos representativos e intente inferir las convenciones. Las que identifique correctamente son las obvias; las que se pierda son las que necesitas documentar.

Trabajar por capas

En vez de dar todo el contexto a la vez, trabajá en capas:

  1. Describí la arquitectura general del módulo que vas a modificar
  2. Compartí solo los archivos directamente relevantes para la tarea
  3. Referencia archivos por nombre cuando necesitas mencionar dependencias sin incluirlas
  4. Si la IA necesita ver algo más, lo pedirá

RAG para equipos grandes

Para equipos que trabajan sobre codebases de 1M+ líneas, el enfoque más robusto es RAG (Retrieval Augmented Generation): los embeddings del codebase completo se almacenan en un vector store, y cada query recupera automáticamente los fragmentos más relevantes.

Herramientas: llamaindex, langchain con ChromaDB o Pinecone, o soluciones de código privado como Sourcegraph Cody.

ℹ️ Monorepos

En monorepos, el contexto más útil generalmente está en el package o módulo donde trabajas, no en el repo completo. Configura las herramientas para indexar el scope correcto, no el monorepo entero.

Cuando la IA no puede ayudar

Con código legacy muy antiguo o con frameworks oscuros (sistemas COBOL migrados, frameworks internos sin documentación pública), la IA tiene poca información de entrenamiento. En estos casos es una herramienta de acompañamiento —puede ayudarte a pensar en voz alta— pero el conocimiento del sistema tiene que venir de los humanos del equipo.
