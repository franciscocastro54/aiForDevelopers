<p>La pregunta no es "¿debo usar IA?" sino "¿para qué tipo de tarea tiene sentido usarla aquí?". Tener un marco de decisión explícito evita tanto el uso irreflexivo como la abstinencia innecesaria.</p>

<h2>La matriz: tarea × riesgo × reversibilidad</h2>
<p>Evaluá tres dimensiones antes de decidir cuánta autonomía darle a la IA:</p>
<ul>
  <li><strong>Claridad de la tarea:</strong> ¿el criterio de éxito es objetivo y verificable?</li>
  <li><strong>Riesgo de error:</strong> ¿un error en producción es catastrófico o recuperable?</li>
  <li><strong>Reversibilidad:</strong> ¿puedes deshacer el resultado si está mal?</li>
</ul>

<h2>Cuándo es subóptimo NO usar IA</h2>
<p>El otro extremo es igualmente real: hay tareas donde no usar IA es desperdiciar tiempo sin ningún beneficio de aprendizaje o control.</p>
<ul>
  <li><strong>Boilerplate predecible:</strong> CRUD genérico, DTOs, serialización, migraciones de schema rutinarias</li>
  <li><strong>Tests de happy path:</strong> generar el primer borrador de tests para una función bien definida</li>
  <li><strong>Refactors mecánicos:</strong> renombrar, reorganizar imports, convertir de callbacks a async/await</li>
  <li><strong>Traducciones de código:</strong> pasar de Python a TypeScript, de SQL procedural a ORM</li>
  <li><strong>Documentación técnica:</strong> JSDoc, OpenAPI specs desde código existente</li>
  <li><strong>Mensajes de commit y descripciones de PR</strong></li>
  <li><strong>Primer borrador de ADRs</strong> (Architecture Decision Records) basados en una decisión ya tomada</li>
</ul>
