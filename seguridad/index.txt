<p>Cuando enviás código a una API de IA, ese texto sale de tu infraestructura y entra al proveedor. Dependiendo del plan, puede usarse para entrenamiento, puede ser revisado por humanos, puede quedar almacenado por períodos variables. La mayoría de los desarrolladores no piensan en esto cuando pegan código en una ventana de chat.</p>

<h2>Qué nunca compartir</h2>

<h3>Secretos de cualquier tipo</h3>
<ul>
  <li>API keys, tokens de acceso, contraseñas</li>
  <li>Certificados y claves privadas</li>
  <li>Cadenas de conexión a bases de datos</li>
  <li>Variables de entorno con valores reales</li>
</ul>
<p>Esto parece obvio, pero los errores ocurren cuando pegás un archivo de configuración "para dar contexto" sin notar que tiene credenciales hardcodeadas.</p>

<h3>Datos personales (PII)</h3>
<ul>
  <li>Dumps de base de datos con datos reales de usuarios</li>
  <li>Logs de producción con información personal</li>
  <li>Resultados de queries con datos no anonimizados</li>
</ul>
<p>Si necesitás debug con datos reales, anonimizalos primero. La IA no necesita ver nombres o emails reales para ayudarte con un bug de SQL.</p>

<h3>Código bajo NDA o acuerdo de confidencialidad</h3>
<p>Si tenés un contrato con un cliente que prohíbe compartir el código con terceros, un proveedor de IA es un tercero. La pregunta "¿cae dentro del NDA?" tiene que responderse antes de pegar código, no después.</p>

<h2>Zero data retention y modos enterprise</h2>
<p>Los proveedores principales ofrecen configuraciones que cambian qué pueden hacer con tus datos:</p>
<ul>
  <li><strong>Zero data retention (ZDR):</strong> los inputs y outputs no se almacenan después de procesar el request. Disponible en planes enterprise de Anthropic y OpenAI.</li>
  <li><strong>Opt-out de entrenamiento:</strong> los datos no se usan para entrenar modelos futuros. En planes pagos generalmente está habilitado por defecto.</li>
  <li><strong>Procesamiento en región específica:</strong> para cumplimiento con GDPR u otras regulaciones de residencia de datos.</li>
  <li><strong>On-premise / VPC:</strong> algunos proveedores permiten deployar el modelo en tu infraestructura. Elimina el riesgo de transferencia de datos.</li>
</ul>
<p>Verificá siempre el plan y los términos actuales del proveedor — las políticas cambian.</p>

<h2>Datos que parecen inocentes pero no lo son</h2>
<p>La información sensible no siempre es obvia:</p>
<ul>
  <li><strong>Estructura de base de datos:</strong> puede revelar modelo de negocio o datos que recopilas</li>
  <li><strong>Nombres de features internas:</strong> pueden dar pistas sobre roadmap de producto</li>
  <li><strong>Arquitectura de seguridad:</strong> exponer cómo autenticás puede ser útil para un atacante</li>
  <li><strong>Logs con IPs:</strong> pueden identificar usuarios o revelar infraestructura</li>
</ul>

<h2>Flujo recomendado para código sensible</h2>
<ol>
  <li>Identificá si el archivo o snippet cae en alguna categoría sensible</li>
  <li>Si tiene credenciales: eliminá antes de compartir, nunca después</li>
  <li>Si tiene PII: anonimizá (reemplazá con datos ficticios del mismo formato)</li>
  <li>Si está bajo NDA: verificá con legal o elegí un modelo local (Ollama + Llama, por ejemplo)</li>
  <li>Si tenés dudas: usá un modelo local para esa conversación específica</li>
</ol>
