Seguridad

Qué no compartir con la IA

Secretos, PII, código bajo NDA y consideraciones contractuales y de privacidad.

Cuando enviás código a una API de IA, ese texto sale de tu infraestructura y entra al proveedor. Dependiendo del plan, puede usarse para entrenamiento, puede ser revisado por humanos, puede quedar almacenado por períodos variables. La mayoría de los desarrolladores no piensan en esto cuando pegan código en una ventana de chat.

Qué nunca compartir

Secretos de cualquier tipo

  • API keys, tokens de acceso, contraseñas
  • Certificados y claves privadas
  • Cadenas de conexión a bases de datos
  • Variables de entorno con valores reales

Esto parece obvio, pero los errores ocurren cuando pegás un archivo de configuración "para dar contexto" sin notar que tiene credenciales hardcodeadas.

Datos personales (PII)

  • Dumps de base de datos con datos reales de usuarios
  • Logs de producción con información personal
  • Resultados de queries con datos no anonimizados

Si necesitás debug con datos reales, anonimizalos primero. La IA no necesita ver nombres o emails reales para ayudarte con un bug de SQL.

Código bajo NDA o acuerdo de confidencialidad

Si tenés un contrato con un cliente que prohíbe compartir el código con terceros, un proveedor de IA es un tercero. La pregunta "¿cae dentro del NDA?" tiene que responderse antes de pegar código, no después.

🚨 Riesgo contractual real

Algunas cláusulas de NDA son amplias y podrían interpretarse como violadas al enviar código a una API de IA. Si tenés dudas, consultá con el área legal o con quien redactó el contrato antes de usar IA con ese código.

Zero data retention y modos enterprise

Los proveedores principales ofrecen configuraciones que cambian qué pueden hacer con tus datos:

  • Zero data retention (ZDR): los inputs y outputs no se almacenan después de procesar el request. Disponible en planes enterprise de Anthropic y OpenAI.
  • Opt-out de entrenamiento: los datos no se usan para entrenar modelos futuros. En planes pagos generalmente está habilitado por defecto.
  • Procesamiento en región específica: para cumplimiento con GDPR u otras regulaciones de residencia de datos.
  • On-premise / VPC: algunos proveedores permiten deployar el modelo en tu infraestructura. Elimina el riesgo de transferencia de datos.

Verificá siempre el plan y los términos actuales del proveedor — las políticas cambian.

Datos que parecen inocentes pero no lo son

La información sensible no siempre es obvia:

  • Estructura de base de datos: puede revelar modelo de negocio o datos que recopilas
  • Nombres de features internas: pueden dar pistas sobre roadmap de producto
  • Arquitectura de seguridad: exponer cómo autenticás puede ser útil para un atacante
  • Logs con IPs: pueden identificar usuarios o revelar infraestructura

Flujo recomendado para código sensible

  1. Identificá si el archivo o snippet cae en alguna categoría sensible
  2. Si tiene credenciales: eliminá antes de compartir, nunca después
  3. Si tiene PII: anonimizá (reemplazá con datos ficticios del mismo formato)
  4. Si está bajo NDA: verificá con legal o elegí un modelo local (Ollama + Llama, por ejemplo)
  5. Si tenés dudas: usá un modelo local para esa conversación específica

💡 Modelos locales como alternativa

Para código altamente sensible, herramientas como Ollama (con Llama 3.1, Mistral, CodeLlama) permiten correr modelos completamente en tu máquina. La calidad es menor, pero el código nunca sale de tu infraestructura.
